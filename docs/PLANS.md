# ExecPlan 执行计划规范

执行计划（ExecPlan）是用于管理复杂、多小时编码任务的设计文档。它是一个"活文档"，在实施过程中持续更新。

## 核心原则

每个 ExecPlan 必须：

- **自包含**：包含实施所需的所有知识和指令
- **是活文档**：随着进展和发现持续修订
- **支持新手**：让不了解项目的人也能端到端实施
- **可验证**：产生可演示的工作行为，而不仅仅是代码变更

## ExecPlan 结构

```markdown
# <简短、行动导向的描述>

本 ExecPlan 是一个活文档。`进展`、`意外发现`、`决策日志`、`结果回顾`等章节必须随工作推进保持更新。

## 目的 / 全局视角

用几句话解释这次变更后用户能获得什么，以及如何看到它在工作。

## 确认状态

- [ ] **待用户确认** — 创建完 ExecPlan 后，将目标、影响范围、关键决策展示给用户，等用户确认后再开始执行。
- [ ] 用户已确认，开始执行

## 进展

使用带复选框的列表来总结步骤。每个停止点都必须记录，即使需要将部分完成的任务拆分为"已完成"和"剩余"两部分。

- [x] (2025-10-01 13:00Z) 示例已完成步骤。
- [ ] 示例未完成步骤。

## 意外发现

记录实施过程中发现的意外行为、bug、优化或见解。

- 发现：...
  证据：...

## 决策日志

记录工作过程中做出的每个决策：

- 决策：...
  理由：...
  日期/作者：...

## 结果回顾

在主要里程碑或完成时总结结果、差距和经验教训。

## 上下文和方向

像读者一无所知一样描述与此任务相关的当前状态。用完整路径命名关键文件和模块。

## 工作计划

以叙述方式描述编辑和添加的顺序。对于每个编辑，命名文件和位置。

## 具体步骤

陈述要运行的确切命令以及在何处运行。

## 验证和验收

描述如何启动或运行系统以及观察到什么。将验收表述为行为，包括具体的输入和输出。

## 产物和笔记

包含最重要的输出、差异或片段作为示例。

## 接口和依赖

命名要使用的库、模块和服务及其原因。
```

## 活文档维护

- 随着做出关键设计决策，更新计划以记录决策及其背后的思考
- 所有决策记录在 `决策日志` 部分
- 当发现意外情况时，在 `意外发现` 部分捕获
- 如果在实施过程中改变方向，记录原因
- 在完成主要任务或整个计划时，编写 `结果回顾` 条目

---

遵循以上指导，一个无状态的 Agent——或人类新手——可以从头到尾阅读 ExecPlan 并产生可工作、可观察的结果。这就是标准：自包含、自给自足、引导新手、聚焦成果。
