# 核心设计信念

本文档记录指导系统设计的核心原则和信念。

## 设计原则

### 1. 简洁优于复杂

> 简单的解决方案更容易理解、维护和扩展。

- 优先选择直接的解决方案，而非过度工程化
- 避免过早抽象，等待模式自然浮现
- 代码应该易于阅读，而不仅仅是易于编写

### 2. 明确优于隐式

> 显式地表达意图，减少"魔法"行为。

- 配置优于约定（在合理范围内）
- 命名应该清晰表达用途
- 避免全局状态和副作用

### 3. 可测试性是第一公民

> 如果无法测试，那就是设计有问题。

- 设计时考虑如何测试
- 依赖注入，便于 mock
- 保持纯函数，隔离副作用

### 4. 渐进式增强

> 先让核心功能工作，再逐步添加高级特性。

- 核心路径优先
- 优雅降级，而非中断
- 不要一次引入所有依赖

### 5. 文档是代码的一部分

> 代码告诉你"怎么做"，文档告诉你"为什么"。

- 重要决策必须记录
- API 必须有文档
- 示例胜过千言万语

## 项目特定原则

### 工具独立性

> 每个工具是独立的——不在工具间引入代码依赖。

- 每个工具拥有自己的依赖、配置和 lifecycle
- 共享代码通过复制或提取到独立 package（需明确决策）
- 工具目录自包含，可单独迁移或删除

### 技术多样性

> 接受并拥抱多语言/多技术栈的现实。

- 小脚本用 Shell，HTTP 服务用 Node.js，数据处理用 Python
- 不存在"首选语言"——选择最适合任务的工具
- 不强制统一构建系统或 CI 流程

### 多 Artifact 交付

> 一个工具可以有多种交付形态。

- 工具的源码和 artifact 声明在 `tools/<name>/` 内
- 支持的 artifact 类型：`cli`（命令行）、`chrome-extension`（浏览器扩展）、`web`（静态站点）
- 不同 artifact 互不干扰——Chrome 扩展和 CLI 安装可以共存
- 构建产物统一输出到 `dist/<name>/`，不提交到 Git

## 反模式

以下是我们明确避免的模式：

| 反模式 | 为什么避免 |
|--------|-----------|
| 工具间的隐式依赖 | 破坏独立性，增加维护负担 |
| 过度抽象共享代码 | 小工具集合不需要 framework |
| 统一技术栈 | 限制工具选择的灵活性 |
| 无 README 或无 .tool.yml 的工具目录 | 别人（和未来的你）无法快速上手，构建系统无法发现 |
| 工具缺少 artifact 声明 | 构建系统不知道要产出什么 |
| 巨型单文件脚本 | 超过 300 行应考虑拆分 |

## 技术偏好

| 偏好 | 原因 |
|------|------|
| Node.js + TypeScript 胜过纯 JavaScript | 类型安全，更好的 IDE 支持 |
| 纯静态前端 胜过 SPA 框架 | 工具项目前端足够简单 |
| Go 或 Python 胜过 Java（CLI 工具） | 更快启动，更低资源消耗 |
