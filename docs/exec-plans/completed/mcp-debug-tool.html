02

关键决策

单文件 Shell 脚本，不拆多文件
按「最小实现」原则。对 ~200 行的调试工具来说，single-file 足够清晰。未来如果 SSE / Streamable 各自超过 150 行再考虑拆分。
依赖 curl + jq，不引入额外包管理器
curl 处理 HTTP / SSE，jq 格式化 JSON。两者都是 macOS 自带或一键安装。无 jq 时降级输出原始 JSON。
Streamable 先实现，SSE 后实现
Streamable 是简单请求-响应模式，SSE 需要管理长连接和异步响应解析。先跑通简单的，再攻克复杂的。
不做 session 复用，每次调用独立握手
这是一个调试工具，不是 SDK。每次调用都完整走 initialize → initialized → action → 清理，确保状态干净。
03

传输协议数据流

两种传输协议的交互模式完全不同：Streamable 是标准 HTTP 请求-响应；SSE 需要维持长连接并通过事件流交换消息。

Streamable: 标准 HTTP 请求-响应，session ID 在响应头中

SSE: 长连接 + 异步响应，需要后台进程管理和临时文件

05

CLI 接口设计

参数简洁，位置参数，不需要 flag 解析。

Usage
# List tools on Streamable endpoint
./mcp-debug.sh streamable http://host/mcpserver1/streamable/mcp list

# List tools on SSE endpoint
./mcp-debug.sh sse http://host/mcpserver1/sse list

# Call a tool on Streamable endpoint
./mcp-debug.sh streamable http://host/mcpserver1/streamable/mcp call tool_name '{"key":"val"}'
JSON-RPC request body (auto-generated)
// tools/list
{
  "jsonrpc": "2.0",
  "id": 1010,
  "method": "tools/list",
  "params": {}
}

// tools/call
{
  "jsonrpc": "2.0",
  "id": 1011,
  "method": "tools/call",
  "params": {
    "name": "<tool_name>",
    "arguments": {}
  }
}
06

风险与对策

风险
级别
对策
SSE 响应解析脆弱：JSON-RPC 响应可能跨多个 SSE data 行，或与其他事件混杂。
HIGH
按 JSON-RPC id 精确匹配；累积 data 行直到完整 JSON 对象；设置 10s 超时防止无限等待。
用户环境无 jq 导致 JSON 输出不可读。
LOW
启动时检测 jq 是否存在；不存在时输出原始 JSON 并提示 brew install jq。
