03

关键决策

Decision 1
保持纯 vanilla JS，不引入 CodeMirror
Chrome 侧边栏宽度仅 ~350px，CodeMirror 完整功能无法充分利用；引入外部库需要构建步骤，破坏当前零依赖架构。textarea + 行号覆盖层方案在此宽度下体验已足够。
被放弃的方案：引入 CodeMirror 6 提供完整代码编辑体验。放弃原因：需要 npm + bundler，增加 ~500KB，与项目"零构建步骤"理念冲突。
Decision 2
Tree 模式保留隐藏的源 textarea 作为数据单一真相源
所有操作（内联编辑、排序、撤销）都修改 textarea 值，然后触发 tree 重渲染。undo/redo 历史栈天然基于 input 事件的值快照，不需要额外的状态管理层。
被放弃的方案：用 JS 对象作为数据源，textare 和 tree 都从对象派生。放弃原因：需要维护双向同步逻辑，undo/redo 需要单独的状态栈，增加复杂度。
Decision 3
保持现有 IIFE 模块模式，不拆分为多个文件
json-editor.js 虽然会增长到 ~600 行，但仍是一个 cohesive 的模块。拆分为 tree.js / code.js / toolbar.js 需要全局变量通信或事件总线，引入不必要的复杂度。600 行的单文件在 vanilla JS 项目中是可维护的。
04

核心架构

Tree 模式的数据流：textarea 是唯一数据源，tree 是只读渲染。编辑通过临时 input 修改 textarea 值后触发重渲染。

Undo/Redo 历史栈结构
// Module-level state
var history = {
  stack: [],    // [snapshot1, snapshot2, ...]
  index: -1,      // current position
  maxSize: 100
};

function pushSnapshot(value) {
  // Discard redo history after index
  history.stack = history.stack.slice(0, history.index + 1);
  history.stack.push(value);
  if (history.stack.length > history.maxSize)
    history.stack.shift();
  else history.index++;
}

function undo() {
  if (history.index > 0) {
    history.index--;
    return history.stack[history.index];
  }
  return null;
}

// Click handler on tree node value
function startEdit(el, path, currentValue) {
  var input = document.createElement('input');
  input.value = currentValue;
  input.className = 'tree-inline-input';

  // Replace text with input
  el.innerHTML = '';
  el.appendChild(input);
  input.focus();
  input.select();

  input.addEventListener('keydown', function(e) {
    if (e.key === 'Enter')  commitEdit(path, input.value);
    if (e.key === 'Escape') cancelEdit();
  });
}

05

风险 & 缓解

风险
严重度
缓解措施
textarea ↔ tree 数据同步不一致：内联编辑后 JSON.stringify 可能改变格式（key 顺序、数字精度），导致用户困惑。
MED
编辑后始终 beautify（2-space indent）；数字保持原始精度（不强制 number → string 来回）；已在路径上精确定位，不整体替换。
Code 模式行号与 textarea 滚动不同步（不同 OS/字体的 line-height 差异）。
LOW
用 textarea.scrollTop 驱动行号 div 的 scrollTop，行高用 parseInt(getComputedStyle(textarea).lineHeight) 动态计算。
Compare 结构化 diff 对大型 JSON（>10K 行）的性能问题。
LOW
递归 diff 本身就是 O(n) 的 key 遍历；如遇性能问题，可在 diff 前添加深度限制（max 50 levels），并显示 "truncated" 警告。实际使用中侧边栏宽度下的 JSON 通常较小。

验证清单

在 Chrome 中加载扩展后逐条验证。✅ = 通过，❌ = 失败需修复。

1. Tree 内联编辑 点击 Tree 中任意值的文本 → 出现 input → 修改值 → Enter → tree 更新且源 JSON 同步
2. 节点操作 hover 任意 tree 节点 → 看到 Add/Remove/Duplicate 按钮 → 点击各按钮均正确修改 JSON 结构
3. 类型转换 点击值的类型标签 → 值在 string ↔ number ↔ boolean ↔ null 间循环切换
4. Code 模式行号 切换到 Code 标签 → 看到左侧行号 → 滚动 textarea 行号同步滚动 → 输入无效 JSON 显示错误位置
5. Sort keys 点击 Sort 按钮 → 所有 object key 递归按字母排序 → tree 重渲染反映新顺序
6. Undo / Redo 修改 JSON → Ctrl+Z 恢复修改前 → Ctrl+Shift+Z 重做修改 → 新编辑丢弃 redo 历史
7. Download 点击 Download 按钮 → 浏览器下载 .json 文件 → 内容与编辑器一致
8. Upload 点击 Upload 按钮 → 选择本地 .json 文件 → 内容加载到编辑器并 beautify
9. Compare 结构化 diff 切换到 Compare → 输入两个不同 JSON → 看到带颜色的 tree diff（非逐行文本 diff）→ Format both / Swap 可用
10. 暗色主题 切换到暗色主题 → Tree / Code / Compare 三个模式所有 UI 正确适配
